## 二、Docker

### 2.1 Docker 是什么

Docker 是一个"打包神器"，它把你的代码、运行环境、依赖库全部打包成一个"集装箱"（镜像），这个集装箱可以在任何安装了 Docker 的机器上运行。

### 2.2 核心概念

| 概念 | 说明 | 类比 |
|------|------|------|
| Dockerfile | 告诉 Docker 如何构建镜像的"说明书" | 菜谱 |
| 镜像（Image） | 打包好的只读模板 | 安装盘（.iso） |
| 容器（Container） | 镜像的运行实例 | 用安装盘启动的电脑 |

### 2.3 Dockerfile 逐行解析

```dockerfile
FROM python:3.12-slim          # 基础镜像：已装好 Python 3.12
WORKDIR /app                   # 设置工作目录
COPY requirements.txt .        # 先复制依赖文件
RUN pip install -r requirements.txt  # 安装依赖
COPY . .                       # 复制所有代码
EXPOSE 8000                    # 声明端口
CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]  # 启动命令
```

**为什么要分两步 COPY？**

Docker 有构建缓存机制。先复制 requirements.txt 安装依赖，再复制代码。这样改代码时，依赖安装那一步会用缓存，构建更快。

### 2.4 常用命令

```bash
docker build -t 镜像名 .     # 构建镜像（不要漏掉末尾的点）
docker run -d -p 8000:8000   # 运行容器（-d 后台运行，-p 端口映射）
docker push 镜像名            # 推送到仓库
docker pull 镜像名            # 从仓库拉取
docker ps                    # 查看运行中的容器
docker stop 容器名            # 停止容器
```

### 2.5 -p 8000:8000 的含义

```
-p 宿主机端口:容器端口
```

把你自己电脑的 8000 端口，映射到容器内部的 8000 端口。浏览器访问 localhost:8000 时，请求会转发到容器里。

## 三、镜像仓库（Docker Hub）

### 3.1 是什么

Docker Hub 是 Docker 官方的公共镜像仓库，相当于 GitHub 之于代码，Docker Hub 之于镜像。

### 3.2 核心操作

```bash
docker login                          # 登录
docker tag 本地镜像名 用户名/镜像名:版本   # 打标签
docker push 用户名/镜像名:版本           # 推送
```

### 3.3 标签（Tag）的作用

```
devtoprod                    → 本地镜像
emeduo/devtoprod:v1          → Docker Hub 上的镜像（带用户名和版本）
```

标签让镜像有了"地址"，任何有 Docker 的机器都可以通过这个地址拉取。

## 四、Kubernetes（K8s）

### 4.1 是什么

Kubernetes 是容器编排工具。当你的应用需要：
- 运行多个副本（负载均衡）
- 自动重启挂掉的容器
- 滚动更新（不停机部署）

就需要 K8s 来管理。

### 4.2 核心对象

| 对象 | 作用 | 类比 |
|------|------|------|
| Pod | K8s 最小运行单元，里面跑一个或多个容器 | 一个"工位" |
| Deployment | 管理 Pod 的副本数量，自动恢复 | "人事部"，保证有足够员工在岗 |
| Service | 在多个 Pod 之间做负载均衡 | "前台"，把请求分发给不同的工位 |

### 4.3 Deployment 配置解析

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: devtoprod
spec:
  replicas: 2                    # 运行 2 个副本
  selector:
    matchLabels:
      app: devtoprod             # 管理哪些 Pod（通过标签匹配）
  template:
    metadata:
      labels:
        app: devtoprod           # 给 Pod 打标签
    spec:
      containers:
        - name: devtoprod
          image: emeduo/devtoprod:v1   # 使用的镜像
          ports:
            - containerPort: 8000      # 容器端口
```

### 4.4 Service 配置解析

```yaml
apiVersion: v1
kind: Service
metadata:
  name: devtoprod
spec:
  type: NodePort                 # 暴露给外部访问
  selector:
    app: devtoprod               # 把流量转发给哪些 Pod
  ports:
    - port: 80                   # Service 端口
      targetPort: 8000           # 转发到容器的端口
      nodePort: 30080            # 外部访问端口
```

### 4.5 流量路径

```
浏览器 → NodePort 30080 → Service（负载均衡）→ Pod 1 或 Pod 2（端口 8000）
```

### 4.6 常用命令

```bash
kubectl apply -f 文件名.yaml   # 应用配置文件
kubectl get nodes              # 查看集群节点
kubectl get pods               # 查看所有 Pod
kubectl get services           # 查看所有 Service
kubectl delete -f 文件名.yaml  # 删除资源
```

## 五、CI/CD（GitHub Actions）

### 5.1 是什么

- CI（持续集成）：每次推代码，自动检查能否正常构建
- CD（持续部署）：构建通过后，自动部署到服务器

### 5.2 工作流文件解读（.github/workflows/deploy.yml）

```yaml
# 触发条件：推送到 main 分支
on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest       # GitHub 提供的免费虚拟机

    steps:
      # 步骤1：拉取代码（相当于 git clone）
      - name: Checkout code
        uses: actions/checkout@v4

      # 步骤2：登录 Docker Hub（用 Secrets 里的加密凭证）
      - name: Login to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}

      # 步骤3：构建镜像并推送
      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: emeduo/devtoprod:latest
```

### 5.3 整个流程

```
git push
  → GitHub 检测到 main 分支有新提交
    → 启动一台临时 Ubuntu 虚拟机
      → git clone 拉代码
      → docker login 登录
      → docker build 构建镜像
      → docker push 推送镜像
    → 销毁虚拟机
```

### 5.4 GitHub Secrets

Secrets 是 GitHub 提供的加密存储功能，用于安全保存敏感信息（密码、token 等）。

在日志中，Secret 的值会显示为 `***`，不会泄露。

## 六、项目使用的工具栈对比（原文 vs 我们）

| 环节 | 原文方案 | 我们的方案 | 学到的概念 |
|------|----------|-----------|-----------|
| 代码管理 | Bitbucket | GitHub | Git 版本控制 |

核心概念完全一致，只是工具不同。

## 七、关键收获

1. Docker 解决的是"在我的电脑上能跑，在你那跑不了"的问题
2. Kubernetes 解决的是"怎么管理成百上千个容器"的问题
3. CI/CD 解决的是"每次部署都要手动操作太麻烦"的问题
4. 这三者组合在一起，就构成了现代云原生应用的基础设施
